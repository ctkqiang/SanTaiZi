# 代码标准

## 1. 命名规范

### 1.1 文件命名
- 模块文件应使用小写字母和下划线命名，例如：`simple_http_flood.go`
- 编译后的二进制文件应与源文件同名，例如：`simple_http_flood`
- 目录名称应使用小写字母，按功能分类，例如：`scanners`、`http`等

### 1.2 变量命名
- 局部变量使用驼峰命名法，例如：`clientCount`
- 全局变量使用大写字母和下划线，例如：`MAX_CONNECTIONS`
- 常量使用大写字母和下划线，例如：`DEFAULT_TIMEOUT`

### 1.3 函数命名
- 函数名使用驼峰命名法，例如：`sendRequest()`
- 导出函数首字母大写，例如：`ExecuteModule()`
- 内部函数首字母小写，例如：`validateInput()`

## 2. 代码风格

### 2.1 缩进和空格
- 使用4个空格进行缩进，不使用制表符
- 操作符两侧应添加空格，例如：`a = b + c`
- 逗号后应添加空格，例如：`func(a, b, c)`
- 大括号应与控制语句在同一行，例如：
  ```go
  if condition {
      // 代码
  }
  ```

### 2.2 代码长度
- 函数长度应控制在50行以内，超过应考虑拆分为多个函数
- 一行代码长度不应超过120个字符，超过应换行
- 适当使用空行分隔不同功能的代码块

### 2.3 注释规范
- 模块应包含模块说明注释，包括功能、作者、版本等信息
- 复杂函数应添加详细注释，说明函数用途、参数和返回值
- 关键代码逻辑应添加注释，提高代码可读性

## 3. 模块结构

### 3.1 模块基本结构
- 模块应包含以下基本部分：
  - 模块信息（名称、描述、作者等）
  - 选项定义（参数、类型、默认值等）
  - 执行逻辑
  - 帮助信息

### 3.2 模块信息输出
- 模块应支持`--info`参数，输出模块信息
- 信息格式应遵循框架要求，包括：
  - name: 模块名称
  - description: 模块描述
  - cnvd: CNVD编号（如有）
  - cve: CVE编号（如有）
  - option: 选项定义（格式：name:type:required:default）

### 3.3 模块执行
- 模块应支持`--execute`参数，执行模块功能
- 执行时应接收命令行参数，格式为：`key=value`
- 执行结果应输出为标准输出，错误信息应包含`error:`前缀

## 4. 错误处理

### 4.1 错误检查
- 所有可能失败的操作都应进行错误检查
- 错误信息应清晰明了，包含足够的上下文信息
- 避免使用panic，应使用返回错误的方式处理异常情况

### 4.2 错误传递
- 错误应从底层函数向上传递，保留原始错误信息
- 使用`fmt.Errorf("failed to do something: %w", err)`包装错误
- 上层函数应根据需要处理或继续传递错误

## 5. 性能优化

### 5.1 资源管理
- 及时释放不再使用的资源，如文件句柄、网络连接等
- 使用defer语句确保资源释放
- 避免内存泄漏和资源耗尽

### 5.2 代码优化
- 避免不必要的计算和重复操作
- 合理使用缓存提高性能
- 优化算法和数据结构

## 6. 测试要求

### 6.1 单元测试
- 模块应包含单元测试，测试关键功能
- 测试覆盖率应不低于80%
- 测试应能独立运行，不依赖外部服务

### 6.2 集成测试
- 模块应能与框架正确集成
- 测试模块在框架中的执行情况
- 验证模块选项的处理和参数传递

## 7. 版本控制

### 7.1 代码管理
- 使用Git进行版本控制
- 提交信息应清晰明了，描述代码变更
- 遵循语义化版本规范

### 7.2 版本标签
- 模块应使用版本标签，例如：`v1.0.0`
- 版本变更应记录在CHANGELOG中
- 重大变更应在版本更新中说明

## 8. 代码审查

### 8.1 审查流程
- 模块代码应经过至少一次代码审查
- 审查应关注代码质量、安全性和性能
- 审查意见应及时处理和反馈

### 8.2 审查标准
- 代码是否符合命名规范和代码风格
- 功能是否完整，是否满足需求
- 是否存在安全漏洞和性能问题
- 测试是否充分，覆盖率是否达标
